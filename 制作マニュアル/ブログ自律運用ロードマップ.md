# ブログ自律運用ロードマップ

更新日: 2026-05-31
対象: コールテン公式サイト `https://www.corduroy.co.jp/blog/`
目的: Masaya の判断・確認・手作業を「公開GO」と「機密・事実確認」まで減らす

## 結論

次にやるべきことは、記事をたくさん書くことではなく、ブログ業務を以下の6工程に分けて、それぞれに自動化ガードと担当エージェントを置くこと。

1. 素材を集める
2. 公開候補を選ぶ
3. 下書きを作る
4. 品質とリスクをレビューする
5. 公開作業を実行する
6. 公開後に測定して改善候補へ戻す

Masaya が毎回働かなくてよい状態にするには、最終的に「週1回、公開候補3本と推奨順を見て、公開する/しないだけ返す」運用に寄せる。

## 人間ゲート

完全自動化しないほうがよい箇所を先に固定する。

Masaya の承認が必要:

- 本番公開、push、デプロイ
- クライアント名・実績・発言・写真を含む記事の公開
- 金額、契約、支援内容などの事実が含まれる記事
- 会社の方針変更に見えるお知らせ

エージェントで進めてよい:

- 素材整理
- 下書き作成
- 内部リンク候補の抽出
- SEO/AIO観点の改善
- OGP/thumbnail の不足検知
- `blog:check` 実行
- PR用の差分整理
- 公開後の測定レポート

## 優先順位

### P0: 先に入れるべき安全基盤

| ID | やること | 目的 | 完了条件 | Masaya負担 |
|---|---|---|---|---|
| P0-1 | GitHub Actions に `npm run blog:check` を追加 | main push前にブログ事故を止める | CIで critical / build / typecheck が落ちたら deploy しない | なし |
| P0-2 | 古い社内マニュアルの正本パスを `~/repos/Corduroy/Public` に更新 | エージェントが古いDropboxパスへ迷わない | `blog-operations-manual.md` などのパスが現行化 | なし |
| P0-3 | `blog:audit --json` を追加 | エージェントが監査結果を機械的に読める | 記事単位で critical / warning / status をJSON出力 | なし |
| P0-4 | `blog:queue` を追加 | 次に直す記事・公開候補を自動で並べる | 公開候補・修正候補・放置下書きが一覧化 | 確認だけ |
| P0-5 | 下書き状態ルールを統一 | noindex長期放置や一覧混入を防ぐ | D/W/P/A の状態が監査で判定できる | 公開GOのみ |

最短で効く順番は P0-1 → P0-3 → P0-4 → P0-2 → P0-5。

### P1: Masayaの確認負荷を減らす運用自動化

| ID | やること | 目的 | 完了条件 | Masaya負担 |
|---|---|---|---|---|
| P1-1 | 素材投入フォルダを1つに固定 | 「どこに材料を置けばいいか」をなくす | `marketing/content/blog-inbox/` を入口にする | 素材を置くだけ |
| P1-2 | 素材から記事ブリーフを自動生成 | 企画整理をエージェント化 | slug/title/読者/CTA/素材/リスクをMD化 | 確認だけ |
| P1-3 | `blog:new` テンプレ生成 | 初期ファイル作成の漏れをなくす | noindex付き下書き、画像フォルダ、チェックリスト生成 | なし |
| P1-4 | 内部リンク候補生成 | 関連記事探しを自動化 | 既存記事から3-5本のリンク候補を出す | 選ぶだけ |
| P1-5 | OGP/thumbnail 不足検知と生成指示 | 画像漏れをなくす | thumb/ogp の有無、推奨プロンプト、配置先を出す | 画像の好み確認だけ |
| P1-6 | PR本文テンプレ生成 | レビューで見るべき点を固定 | 変更内容、監査結果、残リスクがPRに入る | レビューだけ |

P1まで入ると、Masayaは「素材を置く」「公開候補を承認する」「事実だけ見る」役割に寄る。

### P2: 継続的に成果を出す編集部化

| ID | やること | 目的 | 完了条件 | Masaya負担 |
|---|---|---|---|---|
| P2-1 | 週次ブログ運用レポート | 自動で次の打ち手を出す | GSC/GA4/公開記事/下書き/警告を1枚に集約 | 読むだけ |
| P2-2 | 公開候補スコアリング | どれを公開すべきか自動提案 | SEO価値、事業価値、素材充実度、リスクで採点 | 選ぶだけ |
| P2-3 | 記事リライト候補検出 | 古い記事の改善を自動発見 | CTR低下、順位低下、情報鮮度切れを抽出 | 優先承認 |
| P2-4 | SNS展開ドラフト生成 | 公開後の二次利用を自動化 | X/note/LINE向け文面を生成 | 投稿GOだけ |
| P2-5 | クライアント確認パック生成 | 事実確認の往復を短縮 | 相手確認用の短い確認文と該当箇所一覧 | 送るだけ |

P2まで入ると、ブログが「書いて終わり」ではなく、営業・SEO・AIO・SNSの循環に入る。

### P3: 構造移行と完全自律化に近い状態

P3は効果が大きいが、既存記事45本の移行が絡む。新規記事から適用するのが安全。

## 角度別に見るやるべきこと

### 1. 素材管理

現状、素材候補は主に以下にある。

- `Dropbox/Corduroy.inc/marketing/content/blog-drafts/`
- `Dropbox/Corduroy.inc/marketing/blog-drafts/`
- `Dropbox/Corduroy.inc/marketing/content/interviews/`
- `Dropbox/Corduroy.inc/marketing/DATA/gsc/`
- `Dropbox/Corduroy.inc/marketing/DATA/ga4/`

やるべきこと:

- 素材入口を `marketing/content/blog-inbox/` に統一する
- 1記事1フォルダにする
- `brief.md` / `source.md` / `facts.md` / `assets/` / `client-confirmation.md` を標準構成にする
- 音声・MTGメモ・SNSメモ・スクショを同じフォルダへ集約する

自律化できること:

- 新しい素材フォルダを見つけて記事候補化
- 不足情報をチェック
- 記事タイプを推定
- クライアント確認が必要か判定

### 2. 企画・優先順位

やるべきこと:

- 各記事に `business_value`, `seo_value`, `source_readiness`, `risk`, `publish_effort` を付ける
- スコアで公開候補を並べる
- 記事の役割を「検索流入」「事例信頼」「会社理解」「SNS素材」「営業補助」に分ける

優先すべき記事:

1. 事業導線に直結するSEO記事
2. 既に素材が揃っていて公開待ちの下書き
3. クライアント許可が不要な自社思想記事
4. 既存記事から内部リンクしやすい記事
5. 写真・発言・実績確認が必要なクライアント記事

### 3. 下書き生成

やるべきこと:

- `blog:new` で下書きを生成
- 記事タイプ別テンプレートを `blog-templates.md` と同期
- noindex / canonical / JSON-LD / breadcrumb / article-lead を初期状態で入れる
- OGP/thumbnail の保存先も同時に作る

自律化できること:

- 素材から構成案を作る
- タイトル案を5本出す
- descriptionを120字以内で出す
- FAQ案を作る
- 関連リンク候補を出す

### 4. 品質レビュー

やるべきこと:

- `blog:audit --json` を実装してエージェントが読めるようにする
- ブランドボイスlintを追加する
- 禁止表現チェックを追加する
- クライアント実名・数字・断定表現を検出する
- 外部情報を使った記事は出典チェックを必須にする

レビュー観点:

- 事実: 数字、日付、肩書き、サービス内容
- 文体: コールテンらしさ、上から目線になっていないか
- SEO: 検索意図、title、description、見出し
- AIO: リード要約、FAQ、llms.txt
- 導線: CTA、サービスページ、関連記事
- リスク: クライアント確認、個人情報、未公開情報

### 5. 画像・OGP

やるべきこと:

- `thumb.png` と `ogp.png` の存在チェックを `blog:audit` に入れる
- 記事タイプごとに画像スタイルを固定する
- 生成プロンプトを記事メタデータから作る
- `characters.json` からTiki/Teto画像URLを取得するルールを組み込む

自律化できること:

- OGP不足検出
- thumbnail不足検出
- 画像生成プロンプト作成
- alt文生成
- index.astro の thumbnail 候補提案

### 6. 公開作業

やるべきこと:

- `publish-blog.sh` と新しい `blog:audit` のルールを統合する
- `blog:promote <slug>` を作る
- index.astro / llms.txt / sitemap除外 / noindex / dateModified をまとめて更新する
- PRを作成し、Preview URLと監査結果をPR本文に出す

人間ゲート:

- `blog:promote` は本番公開前の変更生成まで
- push / merge / production deploy は Masaya 明示GOが必要

### 7. 公開後

やるべきこと:

- Search Console URL検査を公開後タスクにする
- Bing indexing も同じ流れに入れる
- 公開後SNS文面を生成する
- 既存記事から新記事への逆方向リンクを後追いで提案する
- 7日後/30日後の効果測定を自動化する

自律化できること:

- sitemap反映確認
- title/descriptionの表示確認
- GSC/GA4の週次集計
- 低CTR記事のリライト提案
- 内部リンク追加候補の提案

## 自律稼働の完成形

毎週の流れ:

1. エージェントが素材フォルダと既存記事を読む
2. 下書き候補、修正候補、公開候補をスコア順に出す
3. 必要なら下書きを生成し、画像・内部リンク・FAQ・llms.txt候補まで作る
4. `blog:check` を通す
5. Preview URLとレビュー要点をまとめる
6. Masaya は「公開OK」「保留」「この観点だけ直して」を返す
7. 公開後、GSC/GA4/SNS展開/リライト候補に戻す

Masaya が見るべき最終画面:

- 今週の公開候補 3本
- それぞれの公開価値
- リスク有無
- Preview URL
- 直すべき箇所
- 推奨アクション

## 判断

最短で Masaya の作業を減らすなら、まず P0 を実装する。記事生成そのものよりも、CI・JSON監査・公開候補キューのほうが効果が大きい。

理由:

- 生成はすでにAIでできる
- 問題は「どれを公開していいか」「何が危ないか」「何を直せばいいか」の判断負荷
- ここを機械化すると、Masaya の仕事は編集長判断だけになる
