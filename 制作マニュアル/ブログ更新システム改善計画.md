# ブログ更新システム改善計画

更新日: 2026-05-31
対象: `Public/src/pages/blog/` と `Public/src/pages/blog/index.astro`

## 目的

下書きをためる、公開候補を指定する、AIでブラッシュアップする、レビューで確認する、公開する、という流れを事故なく速く回せる状態にする。

現在のブログは Astro の静的ページで、各記事は `src/pages/blog/*.astro` に個別ファイルとして存在し、一覧は `src/pages/blog/index.astro` に手動で並べている。軽量で自由度は高い一方、記事ファイル・一覧・sitemap・noindex・llms.txt の同期漏れが起きやすい。

## 現状の課題

1. 記事本体と一覧の二重管理
   - 記事ファイルを作っても `index.astro` に追加しないと一覧に出ない。
   - 下書きを一覧に出すか、sitemap に出すか、noindex にするかの判断がファイルごとに分散している。

2. 下書きと公開済みの状態が曖昧
   - `isPublished = false` でも一覧や sitemap との整合が保証されていなかった。
   - 下書き URL がビルドされること自体は Astro の仕様として自然だが、検索エンジンに渡さないガードが必要。

3. AI生成後の残骸やレビュー観点が機械的に検出されていなかった
   - `</content>` や `</invoke>` のような生成残骸が複数記事に残っていた。
   - 日付、構造化データ、内部リンク、llms.txt などの公開前チェックが人間の目視に寄っていた。

4. 公開候補の優先順位を決める材料が足りない
   - 「今どの下書きが公開に近いか」「何を直せば公開できるか」を一覧で見づらい。

## 実装済み: Phase 0

公開事故を防ぐ最低限の監査コマンドを追加した。

```bash
cd /Users/masaya/repos/Corduroy/Public
npm run blog:audit
npm run blog:audit:strict
```

`blog:audit` は全体の監査結果を表示する。`blog:audit:strict` は公開事故につながる critical がある場合だけ失敗する。

現在の critical 条件:

- 一覧にあるのに記事ファイルが存在しない
- 一覧にある記事が `isPublished = false`
- `isPublished = false` なのに `noindex` がない
- `noindex` 記事が sitemap 除外に入っていない
- AI生成残骸が残っている

現在の warning 条件:

- 記事ファイルがあるのに一覧にない
- `index.astro` の日付と記事内の日付がずれている
- `datePublished` / `dateModified` が見つからない
- canonical / BlogPosting / Breadcrumb / FAQ / llms.txt の不足
- 内部ブログリンクが少ない
- TODO / REVIEW 系のマーカーが残っている

## 推奨ワークフロー

### 1. 下書きを作る

下書きは記事ファイルとして作成し、初期状態は以下を守る。

```astro
const isPublished = false;
---
<Layout
  title={title}
  description={description}
  noindex={!isPublished}
>
```

下書き段階では `index.astro` に追加しない。sitemap 除外リストには入れる。

### 2. 公開候補を指定する

公開候補を決めたら、まず監査する。

```bash
npm run blog:audit
```

見るべき順番:

1. critical が 0 か
2. その記事の warning が何か
3. 内部リンク、構造化データ、日付、llms.txt が足りているか

### 3. AIブラッシュアップ

ブラッシュアップ時は以下の順で見る。

1. タイトルと description が検索意図に合っているか
2. 冒頭で読者の課題が明確か
3. コールテンの事例・考え方に寄っているか
4. 関連記事への内部リンクが 3 本以上あるか
5. FAQ が検索者の疑問に答えているか

### 4. レビュー

公開前レビューでは以下を必須にする。

```bash
npm run blog:check
```

中身は以下をまとめて実行する。

```bash
npm run blog:audit:strict
npm run build
npm run typecheck
```

strict 監査だけを素早く見たい場合は以下を使う。

```bash
npm run blog:check:quick
```

レビュー観点:

- 下書きが一覧・sitemap に混ざっていない
- 公開記事が `isPublished = true` になっている
- `datePublished` と `dateModified` が妥当
- OGP / thumbnail が存在する
- llms.txt に載せるべき記事が載っている
- 既存記事からの内部リンク導線がある

### 5. 公開

公開時に行う変更:

1. 記事の `isPublished = true`
2. `index.astro` に記事カードを追加
3. sitemap 除外リストから削除
4. `public/llms.txt` に追加
5. `npm run blog:audit:strict` / `npm run build` / `npm run typecheck` を通す

## 次フェーズ案

自律運用まで含めた優先順位は `制作マニュアル/ブログ自律運用ロードマップ.md` を正とする。

### Phase 1: 公開候補の見える化

`blog:audit` の結果を、下書きごとの公開準備ステータスとして見られるようにする。

候補:

- `npm run blog:audit -- --json` を追加
- critical / warning / missing internal links を記事単位で JSON 出力
- 公開候補の優先順位を AI が読みやすい形式にする

期待効果:

- 「次に公開できそうな記事」をすぐ選べる
- レビューで見るべき箇所が絞れる

### Phase 2: 記事メタデータの単一管理

一覧・sitemap・llms.txt を手動同期しない方向へ寄せる。

候補:

- Astro Content Collections へ移行
- もしくは `src/content/blog/*.mdx` + 共通記事レイアウトに移行
- 既存 `.astro` 記事は段階移行し、新規記事から適用

期待効果:

- タイトル、公開日、公開状態、説明文、タグ、OGP、noindex を frontmatter に集約できる
- 一覧や sitemap の整合性をコードで担保できる

### Phase 3: 下書き生成コマンド

新規下書きをテンプレートから作れるようにする。

候補:

```bash
npm run blog:new -- --slug example-slug --title "記事タイトル"
```

生成するもの:

- 記事ファイル
- OGP / thumbnail の置き場
- 初期 frontmatter
- 公開前チェックリスト

期待効果:

- 毎回の初期セットアップ漏れを減らす
- AI に「このテンプレに沿って埋める」と指示しやすくなる

### Phase 4: レビュー用プレビュー運用

公開前に確認しやすい URL とレビュー観点を固定する。

候補:

- Vercel Preview で公開候補のページを確認
- `blog:audit` の結果を PR コメントに貼る
- 公開候補ごとにレビュー観点を自動生成する

期待効果:

- Masaya が本文品質だけに集中しやすくなる
- 公開作業の属人性が下がる

## 今回の残タスク

critical は 0 まで落とした。残っている warning は、公開事故ではなく品質改善・整理タスクとして扱う。

優先度高:

- `index.astro` と記事内の日付不一致を整理する
- `codex-pet-hatch-guide` と `npo-tools` の日付メタデータを補う
- TODO マーカーが残る記事をレビューする

優先度中:

- 古い記事の内部リンクを 3 本以上に増やす
- llms.txt 掲載対象の基準を決める

優先度低:

- JSON-LD script の `is:inline` hint を全体で整理する
- 既存記事を Content Collections に段階移行する
