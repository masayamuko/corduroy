# サイトリニューアル計画書

## 1. 現状の課題とリニューアルの目的

### 現状の課題
現在、Webサイトは静的なHTMLファイル群で構成されています。そのため、以下の課題が発生しています。
*   **運用コストの増大**: ヘッダーやフッターなどの共通パーツを修正する際、全HTMLファイルを個別に編集する必要がある。
*   **拡張性の欠如**: 新しいページを追加するたびに、共通部分のコードをコピー＆ペーストする必要がある。
*   **SEO・集客施策の限界**: ブログ機能がないため、コンテンツマーケティング（SEO）が実施しにくい。

### リニューアルの目的
*   **保守性の向上**: ヘッダー・フッターなどの共通パーツをコンポーネント化し、一箇所の修正で全ページに反映させる。
*   **ブログ機能の実装**: SEO対策として、記事コンテンツを容易に投稿・管理できる基盤を作る。
*   **パフォーマンス維持**: 現状の高速な表示速度を維持しつつ、モダンな開発環境へ移行する。

---

## 2. 推奨アーキテクチャ：Astro (アストロ)

今回のリニューアルでは、**Astro** というWebフレームワークの採用を強く推奨します。

### Astroを選定する理由
1.  **HTML・CSS中心の開発**: AstroはHTMLに近い構文で書けるため、既存の `index.html` などの資産をほぼそのままコピー＆ペーストで移行できます。ReactやVueなどの複雑なJSフレームワークの知識がなくても扱いやすいです。
2.  **圧倒的なパフォーマンス**: 「アイランドアーキテクチャ」という技術により、必要な部分だけJavaScriptを読み込むため、サイトが非常に高速に保たれます（SEOに有利）。
3.  **コンポーネント指向**: ヘッダー (`<Header />`) やフッター (`<Footer />`) を部品（コンポーネント）として管理できます。
4.  **ブログ機能が標準搭載**: Markdownファイル (`.md`) によるコンテンツ管理が標準でサポートされており、ブログ機能の実装が非常に容易です。

---

## 3. ブログ機能の実装方針

ブログ記事の管理方法として、2つのパターンが考えられます。

### パターンA：Markdownファイル管理（推奨・初期段階）
*   **仕組み**: プロジェクト内の `src/content/blog/` フォルダなどに `.md` ファイルを作成して記事を書く。
*   **メリット**: サーバー不要、コストゼロ、Gitで履歴管理ができる。開発者にとっては最も手軽。
*   **デメリット**: 記事を書くのにコードエディタ（VS Code等）が必要。

### パターンB：Headless CMS (MicroCMSなど) 連携
*   **仕組み**: MicroCMSのような外部の管理画面で記事を書き、APIでサイトに表示する。
*   **メリット**: ブログのような使いやすい管理画面があるため、スマホからでも更新可能。非エンジニアでも更新しやすい。
*   **デメリット**: CMSの初期設定が必要（API連携など）。

**提案**: まずは **パターンA (Markdown)** で構築し、運用が大変になってきたらパターンBへ移行するのがスムーズです。

## 4. リニューアルのステップ

実装は以下のフェーズに分けて進めます。

### Phase 1: 環境構築とベース移行
1.  Astroプロジェクトのセットアップ。
2.  共通レイアウト（`<Layout />`）の作成（`<html>`, `<head>`, `<body>`の外枠）。
3.  共通コンポーネント（`<Header />`, `<Footer />`）の切り出し。
4.  トップページ (`index.html`) の移行とコンポーネント適用。

### Phase 2: 下層ページの移行
1.  サービスページ（Study, System, Advisory）の移行。
2.  各ページで共通コンポーネントを使用するように書き換え。
3.  CSSの整理（グローバルCSSとコンポーネント別CSSへの分割）。

### Phase 3: ブログ機能の実装
1.  ブログ記事用レイアウトの作成。
2.  記事一覧ページ (`/blog`) と記事詳細ページ (`/blog/[slug]`) の作成。
3.  テスト記事の投稿と表示確認。

### Phase 4: SEO・最適化
1.  メタタグ（OGP, Title, Description）の動的生成設定。
2.  サイトマップ (`sitemap.xml`) の自動生成設定。
3.  パフォーマンス計測と最終調整。

---

## 5. 比較：従来のHTML管理 vs 新構成

| 項目 | 現在 (HTML/CSS) | 新構成 (Astro) |
| :--- | :--- | :--- |
| **ヘッダー修正** | 全ファイル修正が必要 | **1ファイル修正で完了** |
| **ページ追加** | HTMLをコピペして作成 | コンポーネントを読み込むだけ |
| **ブログ** | 手動でHTMLを作成 (困難) | **Markdownを書くだけで自動生成** |
| **SEO設定** | 全ファイル手動設定 | **共通設定で自動出力** |

この計画により、日々の更新作業（ヘッダーのお知らせ変更など）の負担が劇的に減り、コンテンツ作成に集中できる環境が整います。
